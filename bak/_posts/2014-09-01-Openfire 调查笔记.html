在 CentOS 中

      rpm -ivh openfire.3.9.1rpm
安装完毕后为在 /opt/ 目录下面生成 openfire 的相关内容。

然后使用 root 权限执行 /opt/openfire/bin 下面的程序：
     # ./openfire.sh

如果执行过程中发生如下错误：
     ./openfire.sh: /opt/openfire/jre/bin/java: /lib/ld-linux.so.2: bad ELF interpreter: 没有那个文件或目录
     ./openfire.sh: line 165: /opt/openfire/jre/bin/java: 成功
则可能是由于 centos 是 64bit 的系统，但是上面没有安装 32bit 的库，可以执行以下命令进行安装：
     yum install glibc.i686
参考：http://stackoverflow.com/questions/8328250/centos-64-bit-bad-elf-interpreter

然后就可以正常启动，浏览器直接运行以下 URL 启动控制台：
     http://localhost:9090

也可以使用以下命令将 openfire 当作 service 运行（也需要 root 权限）：
      service openfire start
（openfire 命令在 /etc/init.d/ 目录下面）该命令的选项还有 stop，restart，status 等。


Openfire 基本设置

首次安装后启动控制台时会要求进行一些基本设置。按照画面进行语言，数据库，管理员密码等设置即可。
其中数据库的设置有多种选择：内置数据库，外部数据库，LDAP。

使用内置数据库的话，openfire 会使用内部的一个库来保存用户情报等相关信息；
在 openfire 安装目录的 bin 目录下有一个 embedded-db-viewer 貌似可以看内置数据库里面的内容。
注意：该 bat（或 sh）里面的 classpath 可能需要修改才能执行成功。
（实际使用了下，没有看到数据库中表的数据，可能是用法有问题。待调查）

使用外部数据库的话（需指定 driver，server 等信息），会根据数据库类型自动创建所需要的表。
表的 schema 可以参考 openfire 安装目录下的  resources\database 下的 sql 文件。

注意：数据库的类型（内置，外部，LDAP）设置完成之后，不可修改。

Web 客户端 Xmppwebchat：
Xmppwebchat 利用了 cometD 来实现浏览器和 web 服务器的长连接。
然后在 web 服务器端使用下面要提到的 openfire java api -- smack 来实现和 openfire 通讯。
简单分析如下：
首先在 web 服务器端建立和 openfire 的连接，然后浏览器端 cometD 使用 cometd.handshake() 去找这个链接。
浏览器如果不明确调用 cometd.disconnect() 去和服务器断开链接，
那么即使浏览器关闭了，web 服务器端和 openfire 的链接还是会始终在的
（是否要等到 session timeout？还是会一直在？貌似是一直在）。
xmppwebchat 在 chat.jsp 被请求的时候，先检查 session 中有没有登录用户的 xmpp 链接。
如果没有就重新链接一个，有的话就直接利用了。
所以只要登录（认证）一次，以后 session 里面有了链接，后面即使刷新页面，和 openfire 之间的连接也不用重建了。
至于 cometD 做 handshake 的时候是如何找到 session 中这个链接的，通过查看其源代码发现：
handshake 的时候并没有找到 web server/openfire 的链接，只是建立 client/web server 的链接。
是在 handshake 之后需要 publish 登录用户名一下，才能找到 web server/openfire 的链接。
从 fiddle 可以看到，在 handshake 的时候，从 web server 会返回一个 clientid，
以后开始的每一个 cometd 请求都会带上这个 clientid。
从 jsp 传递给 chatListener 的 org.cometd.bayeux.server.BayeuxServer 可能比较关键，
它应该就是关联 client/web server 和 web server/openfire 的桥梁。
chatListener 就是一个 org.cometd.server.AbstractService，构造的时候的参数如下：

    public ChatListener(BayeuxServer bayeuxServer, String name) {
        super(bayeuxServer, name);
        ...
    }

其中的 name 应该就是找到 openfire 链接的 key。

Openfire 使用既存数据库

要想让 openfire 使用既存数据库里面的用户信息等相关数据，可以参考官方说明：
     http://www.igniterealtime.org/builds/openfire/docs/latest/documentation/db-integration-guide.html
去修改 xml 配置中的内容。不过网上说可能有问题，不如直接修改 openfire 数据库里面的数据，效果是一样的。
（在 openfire 控制台的系统属性里面貌似也可以修改）

主要核心是修改 ofproperty 表的 provider.user.className，provider.group.className 等几个 provider 的设定值，比如：
     org.jivesoftware.openfire.user.DefaultUserProvider  修改为  org.jivesoftware.openfire.user.JDBCUserProvider
     org.jivesoftware.openfire.group.DefaultGroupProvider  修改为  org.jivesoftware.openfire.group.JDBCGroupProvider
然后添加一些这几个 provider 所需要的属性信息。

以下网站提供了一个可以直接运行的 SQL（当然，需要修改某些设定），用来将 openfire 配置成使用既存数据库：
     http://zoeyyoung.github.io/openfire-xmpp-database.html
注意对于用户（User）情报，既存数据库需要提供 uid，name， email 三个字段。

不过修改 openfire 的配置这种方式貌似只能对应 User 和 Group 的数据，其他的比如用户 Roster（好友信息），
则需要编写程序提供连接既存数据库的 provider。
openfire 中已经提供好各种 provider 的接口了，只要实现对应的接口即可，可以参考下面的例子：
     http://www.igniterealtime.org/builds/openfire/docs/latest/documentation/pluggable-roster-support-guide.html
     http://www.liuhaihua.cn/?p=401
由于使用外部数据库后，用户数据就是只读的了，在 openfire 的控制台无法修改，追加，删除。
所以自己做的 provider 也只要实现查询接口即可。
实现好 provider 后，在 ofproperty 表中加入记录
（openfire 默认使用的是：org.jivesoftware.openfire.roster.DefaultRosterItemProvider）：
     provider.roster.className：自定义的 provider 的 class 名（比如 yyc.roster.CustomRosterItemProvider）
需要注意，由于自定义的 provider 的程序里面的 DB 连接要连接既存的库，所以 provider 里面要重写下 getConnection 方法，
否则就连接到 openfire 的数据库了。例子如下：

private Connection getConnection() throws SQLException {
   return DriverManager.getConnection(JiveGlobals.getProperty("jdbcProvider.connectionString"));
}

然后自定义的 provider 中所有连接数据库的地方都调用该方法获得连接。

关于长连接个数问题

试验发现，从 client 即使发送多个保持长链接的 request：connection.send($pres())，
后一个请求也会使得前一个请求马上被返回，也就是说 openfire 只会同时保留一个长链接。
这个数字也许和 openfire 的下面的设定有关：xmpp.httpbind.client.requests.max
该设定的默认值是 2，而且其文档上说不应该去修改这个值。

关于用户状态问题

每个用户登录到 openfire 的时候，服务器会给其 roster 发送一个标记该用户上线的 message。
但是 openfire 服务器何时认为某个用户是 offline 的呢？
如果该用户在退出时正常发送 disconnect 消息的话，自然没问题，
服务器会给其 roster 发送一个标记该用户下线的 message。
如果登录用户没有发送 disconnect 消息的话，比如直接关闭浏览器，或者关闭机器等等。
测试时发现，这种情况下，原画面 keep 的链接不久后就会结束（当然是失败了），
但是 openfire server 显示该用户还没有 offline，该用户的 roster 也没有收到该用户 offline 的 message，
貌似直到某个 timeout 时间后才变成 offline。

关于使用 attach 跨页保持连接问题

Strophejs 提供了 attach 方法可以在页面间维持同一连接。
其他的 web client 库基本也都提供类似功能，比如 conversejs，
不过也是都需要开发一些程序来实现：https://conversejs.org/docs/html/index.html

要使用 attach 方法去寻找既存连接，需要提供该连接的 jid，sid，rid 三个属性。
     connection.attach(jid, sid, rid, onConnect);
其中 onConnect 方法和连接 openfire 时的基本一致，是一个 handle：     
     function onConnect(status){
          ...
          else if (status == Strophe.Status.ATTACHED) {
               ...  // 和 Strophe.Status.CONNECTED 时一样做
          }
          ...
     }
重点是前面三个参数：
jid 就是登录用户的 jid。
sid 是每个 openfire 的连接的标识符，会在每次 connect 的时候分配一个。
rid 是每次请求的时候自动变更（一般是递增 1）的数字。这里需要传递最后一次被 keep 住的链接的 rid。
如果参数中的 rid 不等于最后一次的 request 的 rid ，那么 attach 会失败，同时连接会被服务器中止。

如果 attach 的时候前一个页面没有断开，那么 attach 也会成功。

需要注意的是，attach 上之后虽然也可以使用和 connect 时一样的方法取得用户的好友一览，
但是并不能接收到 on_roster_change 事件，也就是说，并不能取得当前已经 online 的用户一览，

首先对于群组（group）的数据有一些要求，以下以 openfire 控制台中做数据为例：
（实际外部数据源的数据可参考这边的测试数据）

group 需要开启共享属性：


否则，没有简单的请求可以取得 openfire 服务器上的 group 一览等信息。

开启共享属性之后，在客户端请求 roster 一览的时候，
response 里面对每一个 roster 的 item 都会附带一个 group 信息（如果该 roster 属于某个 group 的话）：

<iq id="HjmA8-2" to="testa@win8vm/Spark 2.6.3" type="result">
  <query xmlns="jabber:iq:roster">
    <item jid="testd@win8vm" name="test D" subscription="both">
      <group>group A</group>
      <group>group C</group>
    </item>
    <item jid="testb@win8vm" name="test B" subscription="both">
      <group>group A</group>
    </item>
    <item jid="testc@win8vm" name="testc" subscription="both"/>
  </query>
</iq>

如上，testd 这个用户属于 group A 和 group C；testb 则只属于 group A；testc 则不属于任何 group。

而且这个附带的 group 信息，只会列出和请求的用户有关的组；
就是说上例中，如果登录用户 testa 不在 group C 里面，而其好友 testd 在 group C 里面，
那么 testd 的 item 下面的 group 列表里面也不会有 group C。

此外，开启“开启联系列表组共享”属性后，这个 group 里面所有的 user 都会变成相互订阅的 roster 了。
也就是说一个组里面所有的用户都是相互好友，并且在 roster 的一览里面可以看到不能删除这种好友关系：


其实偶尔也发现并不是所有的人都是互相为 roster：
比如在 testa 的 roster 里面可能发现有 testc，但是 testc 的 roster 里面没有 testa。
由于还没能定位到具体发生场合及原因，所以没有继续调查这点。

没有开启共享的则不会这样强制其成员成为好友。

那么，根据上述结果，基本可以在用户登录后取到 roster 的同时，
根据其每个 roster 的 group 情报，统计出该用户属于哪几个 group，
每个 group 有哪些用户也是可以统计出来的。

不过有一个例外，就是如果某个 group 里面只有一个用户，
那么在这个用户登录的时候，获得的 roster 情报里面就没有该 group 的记录
（因为他没有好友在这个 group 里面）。

所以这种情况如果存在的话，应该采取下面的方法取得所有的 group 的一览：
发送：
<iq id="r9x1Y-9" type="get">
  <sharedgroup xmlns="http://www.jivesoftware.org/protocol/sharedgroup"/>
</iq>
回应：
<iq id="r9x1Y-9" to="testa@win8vm/Spark 2.6.3" type="result">
  <sharedgroup xmlns="http://www.jivesoftware.org/protocol/sharedgroup">
    <group>group B</group>
    <group>group C</group>
  </sharedgroup>
</iq>
当然这种请求的回应里面，也只是所有开启了“开启联系列表组共享”属性的群组一览。

使用 StropheJS 的场合，可以这样取得：
var iq = $iq({type: 'get'}).c('sharedgroup', {xmlns: 'http://www.jivesoftware.org/protocol/sharedgroup'});
connection.sendIQ(iq, on_get_group);
这样在 on_get_group 方法中就可以解析并取到登录用户所属的所有 group。
每个 group 有哪些用户应该还是和上面一样从 roster 一览的情报里面获取。

需要注意的是，能够取到的 group 情报只有 group 公开的名字。
如果还同时需要 id，那么最简单的方法可以考虑把 group 公开的名字按照固定格式设置（比如 group A{groupa} 这样的格式），
client 端取得名字后进行解析。

*** 也可以考虑自定义 plugin 来实现取得所有 group 的功能。此时就没必要开启共享属性了。
*** 比如，登录用户发送 message 到 getallgroup@groupinfo.win8vm，（其中的 groupinfo 叫做 subdomain，是插件的属性）
*** 就返回该用户的所属群组，以及各个群组的人员一览。
*** 可以参考接下来提到的 broadcast 插件。


关于群聊

接下来看如果在一个 group 里面发送 message（就是群聊）。
当然最简单的想法，当某个用户在一个 group 里面说话的时候，
可以根据 group 里面有哪些用户，逐一发送普通的 message 到所有的组内成员，貌似也是可以实现的。
但是问题就是，这样的做法效率低下，并且接收方无法确认收到的 message 是某个用户的私聊还是在 group 里面的群聊。

openfire 默认自带了一个 plugin，叫做 broadcast，可直接从 openfire 控制台中的“插件->有效的插件”安装。
它提供了发送消息给多人的功能。里面提供的功能包括发送 message，以及取得用户所属的 group 一览等等。
关于其配置以及使用方式的说明，参考：

关于连接 openfire 服务器的时间点

一般，业务画面会有一个 login 画面，用户输入用户名密码后，发送到 web 服务器进行验证。
验证通过后迁移到主页面。
而和 openfire 服务器的连接，一般也考虑使用该 login 画面的用户名密码情报，
而不是要求用户在某个画面再次提供（输入）一次。

那么，最简单的做法，就是在业务 login 画面进行用户验证的同时，连接 openfire 服务器。
但是这么做，可能有几个问题：
1.比较耗时。
     连接 openfire 的验证过程需要发送多个 request，占用一定时间，可能会影响用户体验。
2.只能做 connection 操作。
     如果要一直做到取得用户 roster 的信息，客户端无法判断何时为结束点。
     但是，如果只做 connection 的话，下一个画面在 attach 后就无法获得 roster 状态信息。
     （测试时发现 attach 成功后一直收不到 on_roster_change 事件）

第一个问题貌似没什么好办法。
第二个问题（attach 后就无法获得 roster 状态信息）后来发现几个解决办法：
1.考虑自定义 plugin 去获取初始 online roster 一览，而不是被动等待 on_roster_change 事件。
2.发现只要稍微延迟点发送请求 iq 的 request，就可以获得 roster 的 online 事件了：
          var iq = $iq({type: 'get'}).c('query', {xmlns: 'jabber:iq:roster'});
          setTimeout(function(){connection.sendIQ(iq, onRoster);}, 1000);

   是因为 webkit 不支持？参考：https://github.com/metajack/strophejs/issues/16

关于断开 openfire 服务器连接的时间点

和连接 openfire 类似，断开 openfire 连接的时候（调用 disconnect），也有类似问题需要考虑。
如果只是简单调用 disconnect 后迁移到别的画面：
     connection.disconnect();
     window.location = "login.html";
那么 disconnect 的请求还没来得及发出去，页面可能就迁移了，可能会导致 disconnect 失败。
可以考虑：
     connection.disconnect();
     setTimeout(function(){window.location = "login.html";}, 1000);

关于 Strophe 的 Connection.addHandler 方法

使用 Strophe 的 Connection.addHandler 方法时一定要注意：
传递进去的 callback 函数只有在返回 true 的场合下才会被多次调用！
如果忽略掉返回值那么这个 callback 只会被调用一次。
比如：
     this.connection.addHandler(function(r){that.onRosterChanged(r);}, null, "presence");
要改成
     this.connection.addHandler(function(r){return that.onRosterChanged(r);}, null, "presence");
同时在 onRosterChanged 里面也应该 return true;

关于 attach 后取不到 roster 一览的问题

测试时有时会发现 attach 成功后，接收不到 on_roster 事件，也就是取不到 roster 的一览。
调查发现原因是在 login 画面 connect 的时候做了如下处理：
               if (status == Strophe.Status.CONNECTED) {
                    connection.send($pres());
                    setTimeout(function(){form1.submit();},1000);
               }               
这里的目的是在 login 画面连接上 openfire 后，发送一个维持长连接的请求，然后等待一秒后迁移到下一个画面。
（这里 timeout 时间可以更小，比如 500 就可以了）

实际上，connection.send($pres()) 这个并不需要，注释掉就没有上述取不到 roster 的现象了。
连接上 openfire 后，Strophejs 自动会维持这个连接。

其他小问题

1.当前 login 用户的用户名不能很方便的获得，可以通过从业务侧取得该名称来解决。
2.由于 attach 方法使用的 jid，sid，rid 等参数是通过 cookie 来传递的，
   所以测试时如果同一个 session 如果有多个用户连接 openfire，那么在 attach 的时候会有问题。
   因为如果有多个连接，保存在 cookie 的 jid，sid，rid 里面的值就乱掉了。
3.画面间迁移比较频繁时，可能会有用户状态的变化无法及时反映到画面以及 connection 丢失问题。
4.关于 javascript 中的封装（javascript access private variable from prototype method）的参考：
     http://stackoverflow.com/questions/436120/javascript-accessing-private-member-variables-from-prototype-defined-functions
     http://blog.stchur.com/2011/09/26/true-private-variables-in-javascript-with-prototype/
5.关于 javascript 的 RegExp 使用，如果这样调用：
     var x =  new RegExp("^(.*)\{(.*)}$","ig");
     x.exec("a{b}");
     x.exec("a{b}");
   那么第一个 exec 返回 ["a{b}", "a", "b"]，第二个 exec 会返回 null。
   需要把 RegExp 里面的 flag "g" 去掉：
     var x =  new RegExp("^(.*)\{(.*)}$","i");
   这样多次调用 exec 就都是一样的结果了。
6.关于 HTML 中文件上传控件的 onchange 事件只触发一次问题，需要改变下 bind 方式，参考：
     http://stackoverflow.com/questions/2721250/jquery-change-method-on-input-type-file
   文件上传控件在被点击的时候会自动生成一个 form，
   所以在上传过程中如果用 js 取其 parent，prev 等元素，会取不到想要的。
   上传开始前，结束后，DOM 树是正常的（和 HTML 一致）。
7.关于使用 jquery 给动态生成的控件绑定事件，参考：
     http://stackoverflow.com/questions/6658752/jquery-click-event-doesnt-work-on-dynamically-generated-elements

               $(".a").on('click', ".b", function () {} );
   在执行的时候要求 ".a" 是存在的，“.b" 可以不存在。

        <!-- Name of the server to connect. This property is required. -->

其中 port 和 passwod 对应于 openfire 控制台开启 connection manager 的时候的设定项目内容。
domain 是 openfire 服务器上的 domain 名称，也就是 jid 里面 @ 后面的部分，也就是 openfire 控制台里面的“服务器名称”，必须一致。
否则虽然 connection manager 可以连接上 openfire，但是从客户端登录的时候会报错：认证失败。


hostname 是 openfire 服务器的机器名，这里可以写 ip 或者 hostname。

一般来说，connection manager 和 openfire 是安装在不同的机器，所以 connection manager 默认用的端口也是 5222，5223：
        <socket>
            <default>
                <port>5222</port>
            </default>
            <ssl>
                <port>5223</port>
            </ssl>
        </socket>
这样客户端连接的时候只要把以前直接连接 openfire 时的 server 的  ip 或者 name 改成 connection manager 的就可以了。
当然，如果 connection manager 和 openfire 是安装在同一台机器，那么就不能使用这两个端口了，
因为他们被 openfire 使用了，可以在设定文件中休改成任意没被使用的端口。

由于 web client 要使用 http-bind 连接到 openfire 服务器，所以也要给 connection manager 开启 http-bind。
同样要在上述设定文件中修改（类似于 openfire 控制台开启 http-bind 的设定）：
     <httpbind> 的 <enabled> 为 true（默认是 false）。
     <httpbind> 的 <scriptSyntax> 的 <enabled> 为 true（默认是 false）。

方案就是将“restart 请求”和“resource bind 请求”合二为一。

首先要分析 Strophe.js 来找到发送这两个请求的地方。
1. restart 请求。发送的代码在：

     Line2481
     Line3343
     Line4209：if (data[i] === "restart")

     Line3105：_sasl_auth1_cb
   在该方法中会根据回应结果判断是否继续发送“resource bind 请求”。
2. resource bind 请求。发送的代码在：
      Line3131
   监听该请求的回应的函数是：
      Line 3154 ： _sasl_bind_cb （Line3127 添加的监听）

根据这个分析结果，要把两个请求合并，需要修改 Line 3105：_sasl_auth1_cb 的最后一段 else 里面的做法，直接调用 _sasl_bind_cb 方法。
同时修改发送 restart 请求时的内容，也就是 Line4193 _onIdle 函数里面的 if (data[i] === "restart") 里面那段。
请求内容修改为在 body 里面加一段 iq：
<iq xmlns="jabber:client" type="set" id="_bind_auth_2">
<bind xmlns="urn:ietf:params:xml:ns:xmpp-bind">
     <resource xmlns="urn:ietf:params:xml:ns:xmpp-bind">sid</resource>
</bind>
</iq>

     * Openfire MAY limit the number of simultaneous requests the client makes with the 'requests'
     * attribute. The RECOMMENDED value is "2". Servers that only support polling behavior MUST
     * prevent clients from making simultaneous requests by setting the 'requests' attribute to a
     * value of "1" (however, polling is NOT RECOMMENDED). In any case, clients MUST NOT make more
     * simultaneous requests than specified by the Openfire.
     *
     * @return the number of simultaneous requests allowable.
     */
    public int getMaxRequests() {
        return JiveGlobals.getIntProperty("xmpp.httpbind.client.requests.max", 2);
    }

那么考虑在客户端发送 request 的时候稍微加大点间隔呢？就是重写这个方法：
    Strophe.Bosh.prototype._send = function () {
        clearTimeout(this._conn._idleTimeout);
        this._throttledRequestHandler();
        this._conn._idleTimeout = setTimeout(this._conn._onIdle.bind(this._conn), 100);
    };
把这里的 100 改成 1000 后发现，错误没有了。
但是发现有时还是出错，看来这样是不行的。

结合上述分析，既然修改 openfire 的配置没有起作用，那么是不是 Connection Manager 的这个设置导致的错误呢？
因为不经过 connection manager 的话是没有这个问题的。

于是在 connection manager 的源代码里面 grep：policy-violation，得到结果：
在 org.jivesoftware.multiplexer.net.http.HttpSession 里面有：
    private void checkPollingInterval() throws HttpBindException {
        long time = System.currentTimeMillis();
        if (((time - lastPoll) / 1000) < maxPollingInterval) {
            throw new HttpBindException("Too frequent polling minimum interval is "
                    + maxPollingInterval + ", current interval " + ((time - lastPoll) / 1000),
                    BoshBindingError.policyViolation);
        }
        lastPoll = time;
    }
里面的 maxPollingInterval 就是配置文件的 <polling>5</polling>，默认是 5。
把它改成 0，果然不再有错误了。

可以修改 HttpSession 的代码把几个数字打印到控制台来看看到底是多少：
    private void checkPollingInterval() throws HttpBindException {
        long time = System.currentTimeMillis();
               Log.debug("***"+time);
               Log.debug("***"+lastPoll);
               Log.debug("***"+maxPollingInterval);
        if (((time - lastPoll) / 1000) < maxPollingInterval) {
            throw new HttpBindException("Too frequent polling minimum interval is "
                    + maxPollingInterval + ", current interval " + ((time - lastPoll) / 1000),
                    BoshBindingError.policyViolation);
        }
        lastPoll = time;
    }
编译后放入 lib/cmanager.jar 即可看到效果。

这里面的判断原则，（据推测）应该是规定两个 polling request 之间的间隔不能小于指定值。
所谓 polling request，就是没有实际内容的请求，这样的请求是为了维持 client 的长连接。
其判断逻辑参考：
openfire 里面：
     org.jivesoftware.openfire.http.HttpSessionManager Line305 forwardRequest 方法：boolean isPoll = (elements.size() == 0)
connection manager 里面：
     org.jivesoftware.multiplexer.net.http.HttpSession Line424 createConnection 方法：packetsToBeSent.size() <= 0
两边不是很一致。
而且在 openfire 的 org.jivesoftware.openfire.http.HttpSessionManager  里面，
判断两个 polling request 之间的间隔是否不符合要求，并不是简单地计算时间间隔，
而是有复杂的逻辑，参考其 checkOveractivity 方法中的实现。
而 connection manager 中，上述checkPollingInterval 方法只是简单计算时间差。
那么是不是这里有 bug，造成 policy-violation 错误呢？

抑或还是我们在 StropheJs 的基础上写的共通里面，发送 polling 请求的时候，多发送了？
从目前的结果看，connect 或者 attach 成功后会发送一条 blank request，
然后每次 on roster changed 处理后会发送一条 blank request，这两条的时间间隔有可能比较小。
而且如果有多个 on roster changed 事件的话，他们所发送的 blank request 就会间隔很小。
而这些发送动作都是 strophejs 内部控制的，不使用 connection manager 的时候也是一样，而且没出错。
所以初步结论是：很可能是 connection manager 本身的逻辑错误。
